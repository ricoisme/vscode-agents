# SQL 速查手冊生成技能詳細流程

本文件定義 `sql-cheatsheet-generator` 的工作流程、案例格式、章節覆蓋範圍與品質標準，供生成資料庫方言化 SQL 手冊時使用。

---

## 步驟 1：確認輸入

先確認以下資訊：

1. **資料庫種類**：必填。未提供時必須先詢問。
2. **案例數量**：若未指定，預設為 100。
3. **資料庫版本**：若未指定，使用該資料庫的一般現代主流版本。
4. **目標讀者**：若未指定，預設為初階到中階開發者與資料庫實務使用者。
5. **輸出檔名**：若使用者要求直接建立檔案，優先使用使用者提供的檔名。

### 必問規則

- 若未指定資料庫種類，不得生成泛用版內容。
- 若使用者指定的資料庫模糊，例如「MS SQL」或「SQL」，需要進一步釐清是否指 `SQL Server` 或其他資料庫。
- 若使用者指定非常冷門的資料庫，仍可處理，但只使用該資料庫可確認的能力，不要套用主流資料庫方言。

---

## 步驟 2：蒐集上下文

優先閱讀以下內容：

1. 使用者選取的 SQL 或 schema
2. 目前檔案中的手冊草稿或資料庫文件
3. 工作區中的 migration、DDL、README、ERD、範例查詢

### 上下文應用原則

- 若工作區已有資料表命名規則，沿用既有命名。
- 若已有實際商業領域，例如訂單、會員、商品、庫存，優先用該領域設計案例。
- 若沒有現成上下文，建立一套簡單且一致的示範 schema，不要在案例之間任意改名。

---

## 步驟 3：規劃章節

預設應覆蓋以下主題。章節名稱可調整，但覆蓋範圍不可明顯缺漏：

1. 資料庫與資料表管理
2. 基本查詢與條件篩選
3. 新增、更新、刪除
4. 排序、分頁與限制
5. JOIN 與多表查詢
6. 聚合、分組與 HAVING
7. 子查詢、CTE 與集合操作
8. 字串、日期、時間、數值與條件函式
9. 交易控制、鎖定與一致性
10. 索引、執行效能與查詢優化
11. View、Stored Procedure、Function、Trigger、排程或資料庫原生自動化物件
12. 使用者、權限、安全性與維運操作
13. 匯入匯出、備份還原、批次處理或常見管理工作

### 章節調整規則

- 若資料庫不支援某類能力，例如 SQLite 缺少使用者管理或 Stored Procedure，請替換成更高頻且真實存在的主題，例如 `PRAGMA`、`VACUUM`、`ATTACH DATABASE`、Trigger 或索引策略。
- 不可為了湊滿章節而捏造不存在的管理功能。
- 每章案例數量需大致平衡，不要讓單一章節佔掉大多數案例。

---

## 步驟 4：撰寫案例

每個案例都必須使用以下固定格式：

````md
## 案例 01：建立新資料庫

**使用情境**
[說明什麼情境下會需要這個語法]

**SQL 範例**
```sql
-- 使用符合目標資料庫方言的語法
```

**說明**
[解釋語法用途、關鍵子句、適用時機]

**資料庫注意事項**
- [說明該資料庫的限制、預設行為、版本差異或必要前置條件]

**常見陷阱／實務建議**
- [指出容易犯錯的地方，必要時提供更安全或更穩健的寫法]
````

### 案例撰寫要求

- 案例編號需連續，從 `01` 到指定數量。
- 每個案例都要有完整 SQL，不可只列函式名稱。
- 每個案例都要說明為什麼使用，而不是只給語法。
- 高風險操作要補充風險與驗證建議。
- 效能相關案例至少要觸及索引、掃描成本、排序成本、函式對索引影響、統計資訊或分頁策略之一。
- 交易相關案例至少要提及提交、回滾、鎖定、隔離層級或該資料庫的限制之一。

---

## 步驟 5：處理方言差異

### 方言處理原則

- 主要 SQL 範例必須完全以目標資料庫為準。
- 可以在說明中簡短提及其他資料庫差異，但不能讓主範例混入其他方言。
- 若目標資料庫不支援某功能，不可直接搬用其他資料庫的替代語法。

### 常見處理方式

| 情境 | 處理方式 |
|------|---------|
| 目標資料庫支援該能力 | 直接使用該資料庫語法，補充差異與注意事項 |
| 目標資料庫有替代實作 | 改寫成該資料庫真實存在的等價高頻案例 |
| 目標資料庫完全不支援 | 明確標示不適用，或用其他高頻主題替代 |

### 典型例子

- `SQLite`：不強調使用者管理，改強調 `PRAGMA`、`VACUUM`、交易模式與附加資料庫。
- `PostgreSQL`：可涵蓋 `RETURNING`、`UPSERT`、`CTE`、Window Functions、JSONB。
- `MySQL` / `MariaDB`：可涵蓋 `AUTO_INCREMENT`、事件排程、引擎差異、`EXPLAIN`。
- `SQL Server`：可涵蓋 `TOP`、`IDENTITY`、`MERGE` 的注意事項、交易鎖定提示。
- `Oracle`：可涵蓋 `SEQUENCE`、`DUAL`、PL/SQL、分析函式與權限模型。

---

## 步驟 6：品質驗證與輸出

輸出前逐項檢查：

- [ ] 案例數量是否精確符合要求
- [ ] 每個案例是否都有五個固定欄位
- [ ] SQL 是否符合目標資料庫方言
- [ ] 是否避免重複案例與不一致命名
- [ ] 是否避免產出不存在的函式、保留字或管理功能
- [ ] 是否為繁體中文且格式一致

### 預設輸出結構

```markdown
# SQL 速查手冊 for [資料庫名稱]：[案例數量] 個高頻使用案例

## 適用對象與使用方式

## 目錄

## 第 1 章 ...

## 第 2 章 ...

## 附錄
```

### 檔名規則

- 若使用者指定檔名，直接使用。
- 否則使用資料庫名稱的小寫連字號形式，例如：
  - `postgresql-sql-cheatsheet-100-use-cases.md`
  - `mysql-sql-cheatsheet-100-use-cases.md`
  - `sql-server-sql-cheatsheet-100-use-cases.md`

---

## 應避免事項

- 不要在未確認資料庫種類前產出內容。
- 不要產出泛用 SQL 清單而沒有資料庫方言脈絡。
- 不要為了湊數而重複案例。
- 不要只輸出簡短條列，導致無法實用。
- 不要混用多種資料庫語法。
- 不要忽略高風險操作的警告與前置條件。
